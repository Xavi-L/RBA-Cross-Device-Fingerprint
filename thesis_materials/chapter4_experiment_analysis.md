# 第4章 实验设计与结果分析

本章基于HybridGuard已采集、构造和标注的三端设备指纹数据，对系统有效性进行实验分析。与传统模型评测不同，本文实验重点不是证明某个机器学习算法优于其他算法，而是验证Android Native、WebView宿主和Web前端三端融合是否能够提供有效风险信息，跨层一致性关系是否能够表达规则知识库中的风险判断逻辑，以及在更严格的分组交叉验证下，这类语义特征是否仍具有一定泛化能力。

## 4.1实验环境与数据集说明

### 4.1.1实验环境

本章实验依托HybridGuard原型系统完成。客户端侧包括Android原生采集模块、WebView宿主特征采集模块和Web前端探针模块；后端侧使用FastAPI接收三端数据，按`session_id`完成会话合并，并将合并后的数据保存为JSON和JSONL文件；离线侧使用综合规则知识库驱动本地大模型生成风险标签，再使用轻量模型对风险分进行拟合和消融评估。

风险标签生成阶段使用第三章介绍的综合规则知识库。该知识库整合了跨层一致性规则、核心完整性规则、物理与运行环境规则、攻击场景聚合规则和容错规则。大模型在标注时接收三端设备指纹JSON和规则知识库，输出结构化的`risk_score`与`risk_reason`。因此，本章中的风险分不是人工随意指定的类别标签，而是基于规则知识库对三端环境进行综合分析后得到的连续风险评分。

轻量评估器采用随机森林回归模型，参数与训练侧和消融实验脚本保持一致：

```text
RandomForestRegressor(n_estimators=50, max_depth=10, random_state=42)
```

随机森林在本章中的作用是评估不同特征组合对规则知识库风险标签的拟合能力。本文不将随机森林作为算法创新点，而是将其作为结构化三端指纹的轻量评估器和端侧评分器候选实现。所有消融实验脚本、指标表和图表均保存在`ablation/`目录下，可通过该目录中的脚本复现实验结果。

### 4.1.2数据来源与样本规模

本章主要实验数据来自`training/scored_data.jsonl`，共1323条带风险标签样本。每条样本包含Android Native、WebView宿主和Web前端三类特征，并包含规则知识库驱动的大模型离线标注结果`llm_label.risk_score`。为了更细致地评估泛化能力，分组脚本根据设备字段、安装来源、调试状态、时区、UA和攻击模板等启发式信息，将样本划分为真实物理设备、云真机/测试环境和脚本攻击三类来源。

表4-1 实验样本来源类型与分组规模

| 来源类型 | 样本数 | 分组数 | 含义 |
|---|---:|---:|---|
| `physical_device` | 286 | 65 | 未命中云测和脚本攻击启发式的真实设备类样本 |
| `cloud_device` | 737 | 77 | 安装来源、时区、ADB或云测特征命中测试环境启发式的样本 |
| `script_attack` | 300 | 3 | API重放、无头PC浏览器、廉价模拟器等脚本攻击模板 |

这里需要区分“样本数”和“分组数”。真实样本扩充会围绕同一设备生成多条相似记录，云真机采集也可能包含同一设备或相近设备环境的多条记录，脚本攻击样本则由少量攻击模板批量生成。如果只按行随机切分训练集和测试集，同一设备的扩充样本或同一攻击模板的相似样本可能同时出现在训练和测试中，从而使测试指标偏乐观。因此，后续实验除随机holdout外，还设计了按`group_id`切分的分组交叉验证。

数据来源构成图由`ablation/figures/figure_01_source_distribution.png`给出。整体上，数据集中云真机/测试环境样本数量最多，脚本攻击样本数量固定为300条，真实物理设备类样本数量相对较少。这一分布符合本项目当前阶段的数据采集特点，也决定了本章实验结论更适合解释“三端一致性规则对已有风险标签的表达能力”，而不是声称已经覆盖全部真实业务攻击环境。

| WebView宿主 | 14 | JSBridge注入、桥接延迟、WebView Provider、系统UA、安装来源、debuggable |

## 4.2实验目标、实验设计与评估指标

### 4.2.1实验目标

本章实验围绕三个问题展开。第一，Android Native、WebView宿主和Web前端三端原始特征分别包含多少风险信息，单端、双端和完整三端在风险分拟合上有什么差异。第二，三端之间的跨层一致性关系是否比简单拼接原始字段更能体现系统价值。第三，在避免同源设备或同源攻击模板泄漏后，跨层一致性特征是否仍具有泛化能力。

这三个问题与本文贡献直接对应。本文的重点不是提出新的机器学习模型，而是实现三端融合采集、会话对齐、规则知识库和端侧轻量评分闭环。消融实验正是为了回答：三端系统采到的数据是否真的有用；跨层语义关系是否能成为风险信号；当样本不再随机混合而是按设备和攻击模板分组时，这些信号是否仍然稳定。

### 4.2.2评估指标

由于风险标签是0到100的连续分数，本章以回归指标作为主要评估指标，包括平均绝对误差MAE、均方根误差RMSE和决定系数R2。

```text
MAE = mean(|y_true - y_pred|)
RMSE = sqrt(mean((y_true - y_pred)^2))
R2 = 1 - SSE / SST
```

其中MAE反映模型预测风险分与标签风险分之间的平均偏差，越小表示拟合越接近；RMSE对较大误差更敏感，适合观察未见设备或未见攻击模板上的泛化风险；R2表示模型对风险分方差的解释程度。对于风控评分系统来说，连续分的接近程度比单纯判断高低风险更细致，因为中风险测试机架和高风险模拟器在处置策略上可能不同。

同时，本章将`risk_score >= 80`作为高风险阈值，计算Precision、Recall、F1和Accuracy。高风险F1用于辅助观察模型是否能够识别高风险样本，但不作为唯一结论依据。原因是当前数据中的高风险样本主要来自规则明确的攻击模板，在随机划分下高风险F1容易饱和，因此需要结合MAE、RMSE和分组交叉验证共同分析。

### 4.2.3数据划分方式

本章使用两种数据划分方式。第一种是随机holdout，固定80/20划分，训练集1058条，测试集265条，随机种子为42。该划分用于快速比较不同特征组的信息量，能够直观观察单端、双端和三端特征组合在同一测试集上的误差差异。

第二种是分组交叉验证。该方法按`group_id`切分数据，同一真实设备、同一云测设备或同一脚本攻击模板不能同时出现在训练集和测试集中。由于脚本攻击样本由3个主模板生成，因此采用3折分组交叉验证，使每一折测试集都包含一个未见脚本攻击模板，同时包含一定数量的真实设备组和云测设备组。

表4-4 分组交叉验证各折样本分布

| Fold | `physical_device` | `cloud_device` | `script_attack` |
|---:|---|---|---|
| 1 | 82 rows / 21 groups | 290 rows / 23 groups | 104 rows / 1 group |
| 2 | 128 rows / 21 groups | 236 rows / 28 groups | 97 rows / 1 group |
| 3 | 76 rows / 23 groups | 211 rows / 26 groups | 99 rows / 1 group |

分组划分比随机holdout更严格。随机划分可能让同一设备的扩充样本或同一攻击模板的近似样本同时进入训练和测试，导致模型看似泛化很好；分组验证则要求模型面对未见设备组和未见模板，更接近真实部署时遇到新设备和新攻击变体的情况。Fold构成图见`ablation/figures/figure_02_fold_distribution.png`，holdout与分组MAE对比图见`ablation/figures/figure_03_holdout_vs_grouped_mae.png`。

## 4.3粗粒度三端特征消融实验

### 4.3.1实验目的与分组

粗粒度三端消融实验直接按照特征来源划分Native、WebView和Web三类原始字段，比较单端、双端和完整三端组合对风险分的拟合效果。该实验回答的问题是：如果只使用某一端或某两端原始特征，模型还能在多大程度上拟合规则知识库给出的风险标签。

表4-5 粗粒度三端特征消融实验配置

| 配置 | 使用特征 | 特征数 | 实验目的 |
|---|---|---:|---|
| Native only | Android Native | 33 | 验证底层硬件、系统、电池、传感器等信息量 |
| WebView only | WebView宿主 | 14 | 验证JSBridge、安装来源、Provider、宿主安全信息量 |
| Web only | Web前端 | 18 | 验证UA、WebGL、Canvas、DPR、算力等信息量 |
| Native + WebView | Native + WebView | 47 | 验证底层系统与宿主容器组合 |
| Native + Web | Native + Web | 51 | 验证底层系统与前端运行环境组合 |
| WebView + Web | WebView + Web | 32 | 验证宿主容器与前端环境组合 |
| Native + WebView + Web | 三端完整原始特征 | 65 | 完整三端基线 |

该实验属于“按端删除”的直观消融，适合观察每一端原始字段的信息量。但它还不能直接证明三端融合的核心价值，因为三端原始字段之间存在天然冗余。例如Native设备型号会反映到Web UA，Native屏幕会反映到Web逻辑屏幕和DPR，WebView provider版本也会反映到Web UA的Chrome/WebView版本。因此，本节结果需要结合后续一致性特征实验共同解释。

### 4.3.2随机Holdout结果与分析

随机holdout下的粗粒度消融结果如表4-6所示。

表4-6 粗粒度三端特征随机Holdout结果

| 配置 | 特征数 | MAE | RMSE | R2 | 高风险F1 |
|---|---:|---:|---:|---:|---:|
| Native only | 33 | 1.222 | 2.198 | 0.9933 | 1.000 |
| WebView only | 14 | 1.139 | 1.537 | 0.9967 | 1.000 |
| Web only | 18 | 1.416 | 2.377 | 0.9922 | 1.000 |
| Native + WebView | 47 | 1.129 | 1.515 | 0.9968 | 1.000 |
| Native + Web | 51 | 1.219 | 2.177 | 0.9934 | 1.000 |
| WebView + Web | 32 | 1.131 | 1.537 | 0.9967 | 1.000 |
| Native + WebView + Web | 65 | 1.140 | 1.544 | 0.9967 | 1.000 |

结果显示，所有配置在随机holdout下都取得了较低误差，高风险F1均为1.000。这说明当前数据中的风险标签包含较强规则信号，即使只使用单端字段，模型也能在随机测试集上捕捉到大量判别信息。其中WebView only的MAE为1.139，接近完整三端原始特征；Native + WebView的MAE为1.129，是该表中最低值；完整三端原始特征的MAE为1.140，并不是绝对最优。

这一结果并不表示三端融合没有价值。相反，它说明三端原始字段之间存在较强代理关系。WebView层的JSBridge、安装来源、debuggable、Provider和system UA与规则标签高度相关；Native层的设备型号、系统版本、屏幕和传感器信息会部分反映到Web UA、DPR和WebGL；Web层的UA和渲染特征又会携带宿主环境的间接线索。随机划分下，训练集和测试集可能共享同一设备或模板的相似样本，因此模型可以通过这些代理字段取得较低误差。

因此，粗粒度消融的结论不是“字段越多越好”，而是“按端删除不足以充分体现三端融合价值”。完整三端原始字段并不总是优于双端组合，说明简单拼接更多字段可能引入冗余或轻微噪声。真正需要进一步验证的是：三端之间的语义关系本身是否能作为有效风险信号。下一节的一致性特征消融正是针对这一问题设计。

### 4.3.3分组交叉验证结果与分析

在分组交叉验证下，粗粒度消融结果更加保守。

表4-7 粗粒度三端特征分组交叉验证结果

| 配置 | 特征数 | MAE均值 | MAE标准差 | RMSE均值 | 高风险F1均值 |
|---|---:|---:|---:|---:|---:|
| Native only | 33 | 2.799 | 1.541 | 4.774 | 0.667 |
| WebView only | 14 | 1.541 | 0.435 | 2.643 | 1.000 |
| Web only | 18 | 12.188 | 7.175 | 23.381 | 0.333 |
| Native + WebView | 47 | 2.202 | 0.840 | 3.767 | 1.000 |
| Native + Web | 51 | 6.573 | 3.757 | 12.522 | 0.333 |
| WebView + Web | 32 | 3.053 | 0.994 | 5.100 | 0.978 |
| Native + WebView + Web | 65 | 2.642 | 1.004 | 4.455 | 1.000 |

与随机holdout相比，分组验证下整体误差明显上升，尤其是Web only的MAE达到12.188，RMSE达到23.381。这说明单独Web前端特征对未见设备或未见模板的泛化不足。Web侧字段容易受到UA、平台、WebGL和Canvas模板的影响，在随机划分中可以表现很好，但当测试集包含未见设备组或未见攻击模板时，单独Web信号难以稳定复现规则知识库中的完整风险判断。

WebView only在分组验证中表现较强，MAE为1.541，高风险F1为1.000。这主要与当前数据和规则标签的构成有关：JSBridge注入、安装来源、debuggable、WebView provider、system HTTP agent等字段直接体现宿主真实性和测试环境特征，是规则知识库中的强信号。Native + WebView组合的MAE为2.202，也优于完整三端原始特征的2.642，说明在当前数据集上，宿主容器与Native底层环境组合已经覆盖了大量风险判断信息。

分组实验进一步说明，完整三端原始字段不是简单拼接越多越好。Web层原始字段在未见设备或未见模板上可能引入噪声，而WebView与Native中的强规则信号更稳定。这为后续一致性特征实验提供了动机：与其直接堆叠三端原始字段，不如把三端之间可解释的对齐、互证和矛盾关系显式编码。

## 4.4跨层一致性特征构造与消融实验

### 4.4.1实验目的

粗粒度三端消融只能回答“哪一端字段有信息量”，但不能直接回答“Native、WebView和Web三端之间是否形成可互证的语义关系”。三端融合的价值不应只体现在字段数量增加，而应体现在不同层级之间可以相互校验、相互印证和发现矛盾。因此，本节将三端之间的语义对齐关系编码为显式一致性特征，再评估这些特征对风险分拟合的作用。

一致性特征来源于综合规则知识库中的跨层语义规则，并由`ablation/run_consistency_ablation.py`实现为数值特征。与原始字段相比，一致性特征更接近风控判断逻辑。例如，模型不再直接看到两个字符串，而是看到“Native型号是否在Web UA中得到印证”；不再直接看到两个屏幕尺寸，而是看到“物理屏幕与Web逻辑屏幕乘DPR的误差是否在合理范围内”。

### 4.4.2一致性特征构造方法

本实验共构造38个`consistency_*`特征，分为四组。

表4-8 跨层一致性特征分组与语义

| 分组 | 特征数量 | 语义 |
|---|---:|---|
| Native-Web | 18 | Android底层环境与Web JS暴露环境是否一致 |
| Native-WebView | 8 | Android底层环境与App/WebView宿主是否一致 |
| WebView-Web | 5 | WebView容器与Web前端运行环境是否一致 |
| Tri-layer semantic | 7 | 三端共同参与的核心风险规则与综合一致性分 |

代表性特征如下。

表4-9 代表性一致性特征及含义

| 特征 | 含义 |
|---|---|
| `consistency_native_web_model_ua_strength` | Native设备型号、产品、主板、品牌在Web UA中的宽松匹配强度 |
| `consistency_native_web_android_version_delta` | Native Android版本与Web UA Android版本差异 |
| `consistency_native_web_screen_score` | 物理屏幕、Web逻辑屏幕、DPR的软一致性分 |
| `consistency_native_web_gpu_family_score` | Native硬件族与WebGL GPU族的宽松匹配分 |
| `consistency_native_webview_agent_model_match` | Native设备型号是否出现在WebView system HTTP agent |
| `consistency_webview_web_chrome_major_match` | WebView Provider与Web UA Chrome主版本是否一致 |
| `consistency_tri_layer_sensor_bridge_fail` | 传感器严重缺失或JSBridge未注入 |
| `consistency_tri_layer_failure_count` | 多个一致性检查失败的聚合计数 |

这些特征强调“宽松匹配”而不是字符串完全相等。例如，Native层的`device_hardware=qcom`与WebGL中的Adreno GPU在字符串上不同，但在移动硬件生态上具有一致关系；MediaTek与Mali、PowerVR，Huawei/Kirin与Mali、Maleoon也可以建立类似映射。屏幕一致性也不能直接比较字符串，而需要将Web逻辑像素乘以DPR后与Native物理像素比较，并允许状态栏、导航栏和安全区造成的高度误差。

Tri-layer semantic组是最贴近规则知识库核心判断的一组特征。它不只比较两端字段，而是综合三端信息判断核心完整性、传感器与JSBridge是否失败、manual安装来源是否伴随时区或ADB、ADB是否与满电接线共同出现、官方安装来源与核心完整性是否同时通过，以及多个一致性检查失败的聚合数量。这组特征数量少，但语义浓缩程度高。

`Tri-layer semantic`仅使用7个特征，MAE为1.111，RMSE为1.469，也接近甚至略优于完整原始三端特征。该结果非常重要，因为它说明高质量语义规则并不一定需要大量字段。传感器与JSBridge完整性、manual安装来源与时区或ADB的组合、核心完整性通过情况、失败计数等少量综合特征，就能表达当前规则知识库中的主要风险判断逻辑。

`Raw all + Consistency`和`Raw cleaned + Consistency`没有明显优于`Consistency only`，说明当前标签体系下，显式一致性特征已经覆盖主要风险信息。加入更多原始字段后，模型并未获得显著额外收益，反而可能受到冗余字段和高基数字段影响。`WebView-Web consistency`单独效果较弱，说明只看WebView provider与Web UA、JSBridge和WebView token的关系不足以完整判断风险，仍需要Native层参与。

总体来看，随机holdout结果支持本文的核心观点：三端融合的价值不只是采集更多字段，而是把不同层级之间的语义呼应关系显式建模。字段之间的冗余不是问题本身，关键在于系统能否把这种冗余转化为互证关系和矛盾检测能力。

## 4.5分组交叉验证下的跨层一致性泛化分析

### 4.5.1实验目的

随机holdout虽然便于快速比较不同特征组合，但可能让同一设备扩充样本、同一云真机环境或同一攻击模板生成的样本同时进入训练集和测试集。在这种情况下，模型可能记住了设备或模板的局部特征，而不是真正学到可泛化的跨层语义关系。因此，本节采用分组交叉验证，更严格地评估未见设备、未见云测环境和未见攻击模板上的表现。

分组验证的意义在于检验系统是否过度依赖同源样本记忆。如果某组特征在随机holdout下表现很好，但在分组验证中明显退化，说明它可能更依赖已见设备或模板；如果某组特征在分组验证中仍保持较低误差，则说明其语义关系具有更强泛化价值。

### 4.5.3结果分析

首先，`Consistency only`在随机holdout中略优于`Raw all`，但在分组交叉验证中MAE上升到3.388，不再优于`Raw all`的2.642。这说明并非所有一致性特征都能稳定泛化。38个一致性特征中既包含强语义规则，也包含依赖设备表达方式的宽松匹配规则。当测试集包含未见设备或未见模板时，部分型号、屏幕、GPU、UA关系可能出现新的表达形式，导致整体一致性特征组误差上升。

其次，`Tri-layer semantic`在分组验证中表现最好，MAE为2.281，RMSE为3.358，优于完整原始三端特征`Raw all`。该配置仅使用7个三端语义规则特征，却在更严格设置下取得最低误差，说明核心规则比完整字段拼接更稳定。传感器与JSBridge完整性、manual安装来源与时区或ADB的组合、核心链路通过情况和失败计数等特征，并不依赖具体设备型号字符串，而是直接表达风险场景，因此更容易跨设备和跨模板泛化。

第三，`Native-WebView consistency`的MAE为2.608，也优于`Raw all`的2.642，且高风险F1为1.000。这说明当前数据集中App宿主真实性、JSBridge、安装来源、debuggable和system HTTP agent是重要风险信号。相比Native-Web的型号、屏幕、GPU匹配，Native-WebView关系更直接反映“是否来自预期App宿主”和“是否处于测试/调试环境”，因此在分组验证中更稳定。

第四，`Native-Web consistency`在分组验证中误差较大，MAE达到12.733。这并不意味着Native-Web关系无用，而是说明单独使用这组关系不足以处理未见设备。不同厂商设备在UA、屏幕、WebGL renderer、硬件族和Web API暴露上差异较大，宽松匹配规则在已见设备上有效，但面对未见设备时容易出现覆盖不足。该结果提示后续需要扩大设备覆盖，并进一步完善硬件族、GPU、UA和屏幕规则。

最后，`Raw all + Consistency`没有比`Raw all`更好，说明简单增加特征数量不一定提升泛化能力。当原始字段和一致性字段共同输入模型时，模型可能仍会依赖某些高基数字段或同源样本强相关字段，导致未见分组下误差上升。真正有价值的是经过抽象和筛选的核心三端语义规则，而不是无差别地堆叠全部字段。

因此，本节核心结论是：在更严格的分组交叉验证下，三端语义规则特征仅使用7个特征便取得优于完整三端原始特征的误差表现，说明跨层一致性建模不是单纯依赖同源样本记忆，而是在未见设备和未见攻击模板上仍具有一定泛化能力。

## 4.6特征重要性与跨层规则解释

### 4.6.1实验目的

前几节主要从误差指标说明一致性特征有效。本节进一步分析随机森林在`Consistency only`配置下依赖哪些特征，以验证模型是否确实利用了跨层语义关系，而不是只记忆原始字符串或单端字段。特征重要性结果来自`ablation/consistency_top_feature_importance.csv`，对应图见`ablation/figures/figure_05_consistency_feature_importance.png`。

### 4.6.2重要特征结果

`Consistency only`配置下排名靠前的特征如下。

表4-13 重要一致性特征及风险解释

| 特征 | 重要性 | 解释 |
|---|---:|---|
| `consistency_tri_layer_sensor_bridge_fail` | 0.1518 | 传感器严重缺失或JSBridge未注入，对高风险样本关键 |
| `consistency_native_webview_debug_cleartext_tension` | 0.1017 | debug与明文网络配置共同反映测试或调试宿主特征 |
| `consistency_native_web_model_ua_strength` | 0.0850 | Native设备身份与Web UA是否能相互印证 |
| `consistency_tri_layer_failure_count` | 0.0844 | 多个一致性检查失败的聚合信号 |
| `consistency_webview_web_ua_has_wv_token` | 0.0843 | Web UA是否呈现WebView运行时特征 |
| `consistency_native_web_model_ua_match` | 0.0839 | Native型号是否与Web UA直接匹配 |
| `consistency_tri_layer_manual_timezone_or_adb` | 0.0749 | manual安装来源结合时区或ADB的云机房风险 |

排名第一的`consistency_tri_layer_sensor_bridge_fail`同时涉及Native物理可信性和WebView宿主真实性。传感器严重缺失说明底层设备生态不完整，JSBridge未注入说明页面可能脱离App宿主。二者都是规则知识库中的核心完整性信号，因此对高风险样本具有较强解释力。

`consistency_native_webview_debug_cleartext_tension`和`consistency_tri_layer_manual_timezone_or_adb`反映测试环境和云机房特征。单独的debug、cleartext、manual、ADB或时区异常并不一定高危，但当这些字段组合出现时，就更接近测试机架、云真机或群控环境。特征重要性结果说明模型确实捕捉到了这种多字段组合规则。

`consistency_native_web_model_ua_strength`和`consistency_native_web_model_ua_match`说明设备身份一致性仍然重要。正常WebView UA往往包含Android版本和设备型号，能够与Native层Build信息互相印证；如果Native显示为移动设备，而Web UA显示Windows、Headless或脚本客户端，则风险显著升高。

`consistency_webview_web_ua_has_wv_token`说明Web层是否呈现WebView运行时特征，也是判断页面是否位于预期App容器中的辅助依据。虽然部分厂商UA可能省略`wv` token，因此该规则需要容错，但在本数据集中它仍能提供有价值的宿主环境信号。

### 4.6.3规则解释与系统贡献对应关系

从规则类别看，特征重要性与本文系统设计高度对应。首先是宿主真实性。JSBridge注入、WebView UA标记、WebView provider与Chrome主版本一致性，均用于判断Web页面是否运行在预期App宿主中。该类规则体现了WebView层在三端融合中的独立价值。

其次是设备身份一致性。Native型号、产品、品牌、Android版本与Web UA的宽松匹配，用于判断不同层级是否像来自同一台设备。这类规则直接支撑“跨层语义对齐”思想，避免模型只依赖单个原始字符串。

第三是物理可信性。传感器矩阵、电池状态、ADB、安装来源等特征组合用于区分真实用户设备、云真机/测试机架和模拟器。特别是传感器与JSBridge失败聚合特征，说明高风险样本往往同时破坏底层物理可信性和宿主完整性。

第四是渲染与运行环境一致性。WebGL renderer、GPU family、Headless、SwiftShader、platform和算力耗时等特征用于识别桌面环境、无头浏览器和软渲染模拟器。虽然这些字段在分组验证中需要更多设备覆盖来提高泛化，但它们仍是规则知识库中不可缺少的一部分。

总体而言，特征重要性结果证明模型关注的不是孤立字段，而是传感器、JSBridge、UA、安装来源、调试状态和运行环境之间的语义关系。这与本文系统贡献一致：HybridGuard的核心不是多采几个字段，而是建立Native、WebView和Web前端之间可采集、可对齐、可解释的跨层设备指纹关系。

## 4.7本章讨论与局限性

### 4.7.1实验结论汇总

本章实验可以归纳为四点结论。第一，粗粒度三端消融显示，单端或双端原始字段在随机划分下也能取得较低误差，说明三端原始字段之间存在较强冗余和代理关系。完整三端原始字段不是所有指标上的绝对最优，这并不削弱三端融合的意义，而是说明不同层级对同一设备状态存在可互证的信息表达。

第二，跨层一致性消融显示，显式一致性特征可以达到与完整原始三端特征相近甚至更优的效果。`Consistency only`在随机holdout下优于`Raw all`，`Tri-layer semantic`仅7个特征就接近完整三端原始特征，说明跨层语义比对本身是有效风险信号。

第三，分组交叉验证显示，随机holdout会高估泛化能力。在更严格的分组设置下，`Tri-layer semantic`仍优于`Raw all`，说明核心三端语义规则比完整原始字段拼接更稳定，更能表达规则知识库中的关键风险逻辑。

第四，特征重要性分析显示，模型重点依赖传感器与JSBridge完整性、Native与Web UA匹配、WebView宿主标记、manual安装来源与ADB等跨层语义信号。这进一步说明模型学习到的是规则知识库中的核心关系，而不是单纯依赖某一端孤立字段。

### 4.7.2局限性说明

本章实验仍存在一些局限。首先，当前数据集规模有限，真实物理设备和真实攻击环境覆盖仍不足。虽然数据集中包含1323条带标签样本，但真实设备品牌、系统版本、WebView provider版本和网络环境仍有扩展空间。

其次，高风险攻击样本主要来自模板化构造，包括API重放、无头PC浏览器和廉价模拟器。模板化样本有助于验证规则知识库和一致性特征是否能识别典型风险场景，但仍不能完全代表真实黑产环境中的复杂攻击链路。后续需要采集更多真实攻击或灰产设备样本。

第三，风险标签由规则知识库驱动的大模型离线标注流程生成，虽然具备可解释性，但仍需要人工抽样复核或业务标注进一步验证。综合规则知识库可以作为标签复核和规则迭代的基础，但最终风险标签质量仍取决于规则覆盖范围、样本多样性和人工校准。

第四，本章主要是离线消融实验，没有将端侧运行耗时、功耗、内存占用和大规模真机闭环作为主要实验。端侧评分App已经在系统实现层面完成，但其性能开销和稳定性仍需要在更多设备上进一步评估。

第五，当前分组策略依赖启发式`source_type`和`group_id`构造。该策略能够降低同源样本泄漏，但仍不如采集阶段直接记录真实设备组、云测设备ID和攻击模板ID严谨。后续实验可在数据采集环节记录更明确的分组元数据。

### 4.7.3后续改进方向

后续工作可以从五个方面展开。第一，扩大真实设备覆盖，增加不同品牌、系统版本、WebView内核版本和网络环境下的样本。第二，构造更多跨层矛盾专项样本，例如Android真机搭配Windows UA、屏幕DPR不一致、WebGL软件渲染、JSBridge缺失、WebView provider与UA主版本不一致等。第三，引入人工抽样复核，校准规则知识库和大模型标签，提高风险标签可信度。第四，对端侧评分App进行系统开销测试，包括采集耗时、推理耗时、内存占用、模型代码体积和评分摘要上传成功率。第五，研究更轻量的规则执行或模型压缩方案，使三端语义规则更适合移动端部署。

## 4.8本章小结

本章基于已标注的三端设备指纹数据，围绕三端融合有效性和跨层一致性建模展开实验分析。首先介绍了数据来源、标签分布、采集字段完整性和实验评估方式；随后通过粗粒度三端消融说明Native、WebView和Web三类原始字段均包含风险信息，但原始字段之间存在较强冗余，简单拼接并不能完全体现三端融合价值；进一步通过一致性特征消融说明，将设备型号与UA、Android版本、屏幕DPR、WebView provider、JSBridge、GPU、传感器、安装来源等跨层关系显式编码后，可以得到有效风险信号；最后通过分组交叉验证证明，在避免同源设备和同源攻击模板泄漏后，三端核心语义规则仍具有较好的泛化能力。

实验结果表明，本文系统的关键价值不是提出新的机器学习模型，而是建立Android Native、WebView宿主和Web前端之间可采集、可对齐、可解释的跨层设备指纹关系。跨层一致性建模能够将规则知识库中的风险经验转化为轻量模型可学习的结构化信号，并为端侧风险评分闭环提供实验依据。
