## 1. 执行摘要

### 1.1 总体判断

项目方向是成立的，但必须把原来的线性设想：

~~~text
LLM 分析数据
  -> 自动形成规则库
  -> 把知识库放进 RAG
  -> 调整 RAG 提升模型表现
  -> 做消融
  -> 写论文
~~~

改造成两条隔离、可验证的链路：

~~~text
离线知识构建：
训练折数据 + 官方知识
  -> LLM 提出候选规则
  -> 确定性统计、反例检查、配对攻击验证
  -> 人工审核
  -> 发布版本化知识库和检索索引

在线风险推理：
177 字段三端数据
  -> Schema 校验与规范化
  -> 确定性跨层证据提取
  -> 字段感知知识检索
  -> LLM 六组语义推理
  -> 规则、字段和引用验证
  -> 外部融合与校准
  -> 风险结果和完整决策追踪
~~~

LLM 可以生成候选规则，但不能直接决定某条规则是真值，也不能直接修改正式规则库。RAG 的效果是需要验证的研究假设，不能预设“一定提分”。

### 1.2 推荐论文主线

推荐把论文主线表述为：

> HybridGuard 是一个可溯源、可验证的三端跨层设备完整性推理系统。系统将 Android Native、WebView 宿主和 Web 运行时数据转化为结构化跨层证据，检索官方语义、审核规则与训练内案例，由大模型完成分组语义判断，再通过确定性验证器和外部融合模块输出风险决策，并在未见设备与未见攻击条件下检验泛化能力。

这条主线的研究贡献不是“使用了向量数据库”，而是：

1. 177 字段三端采集数据的规范化与跨层证据表示。
2. 官方知识、确定性不变量、经验规则和案例知识的分层治理。
3. 面向字段和证据组的可审计检索。
4. LLM 候选规则生成、结构化推理与引用验证闭环。
5. 面向重复设备画像、配对攻击和未见攻击工具的防泄漏评估协议。

### 1.3 五条不可破坏的原则

1. **先划分数据，再生成经验规则。** 最终测试组不得参与规则发现、案例索引、Prompt 调整或检索调参。
2. **会话数不等于独立设备数。** 所有规模报告必须同时给出 session 数和 stable profile 数。
3. **官方文档解释字段语义，不提供风险概率和项目阈值。**
4. **LLM 输出候选和语义判断，确定性程序负责字段比较、核心规则和结果校验。**
5. **没有独立攻击真值时，只能讨论框架、教师一致性和可解释性，不能宣称真实攻击检测准确率。**

## 3. 问题定义、威胁模型与结论边界

<a id="RESEARCH-TASK"></a>

### 3.1 推荐任务定义

在没有真实欺诈结果、账户损失或业务拒付标签时，主任务应定义为：

> 三端跨层设备完整性与受控环境操纵检测。

不要直接定义为“欺诈检测”。

建议将标签拆成四层：

1. **环境事实**：物理真机、云真机、云模拟器、本地模拟器、脚本重放等。
2. **操纵事实**：是否执行了受控操纵。
3. **完整性违规**：UA、版本、GPU、屏幕、JSBridge、传感器等哪些关系被破坏。
4. **策略风险**：根据检测概率、攻击成功和业务代价派生的风险分。

risk_score 是策略输出，不应作为最原始的事实真值。

<a id="RESEARCH-CLAIMS"></a>

### 3.2 可以和不能宣称的结论

| 阶段 | 可以宣称 | 不能宣称 |
|---|---|---|
| 当前未标注 V2 数据 | 完成 177 字段采集、Schema、重复画像审计和系统联调 | 攻击检测准确率、RAG 提升 |
| 现有 llm_label 数据 | 教师一致性、结构预验证、分组融合可行性 | 独立安全真值、真实攻击正确率 |
| 配对攻击 pilot | 对特定工具和配置的受控操纵检测 | 未知攻击泛化、真实欺诈效果 |
| 锁定测试集与 OOD 实验 | 未见设备、工具、家族或 provider 上的泛化 | 超出实验覆盖范围的通用安全能力 |
| 有真实业务结果后 | 风险评分与真实业务风险关联 | 在没有业务结果前提前声称欺诈检测 |

<a id="RESEARCH-HYPOTHESES"></a>

### 3.3 研究假设与否证条件

#### H1：显式跨层证据比直接堆叠字段更稳定

假设：

> Native–WebView–Web 的显式关系，在 stable profile 分组评估下，能够比单端特征和 177 字段直接堆叠更稳定地检测受控操纵。

否证条件：

- 提升只存在于随机切分；
- 分组切分后优势消失；
- benign FPR 明显升高；
- 结果主要由机型、provider 或 App 构建版本捷径贡献。

#### H2：官方知识改善语义边界与可审计性

假设：

> 官方知识降低字段语义误解、错误引用和开发配置误报，但不预设一定提高总体检测率。

否证条件：

- 去掉官方知识后引用正确率、边界误报和解释质量都没有变化；
- 模型把官方字段语义错误解释为官方风险阈值；
- 随机错配官方卡片也获得相同结果。

#### H3：经验证的数据规则具有未见攻击泛化价值

假设：

> LLM 在训练组上提出、经程序和人工审核后晋级的经验规则，能够改善未见设备或攻击家族上的覆盖率。

否证条件：

- 只提升已见工具和配置；
- held-out attack family 无增益；
- clean FPR 明显上升；
- 新规则与官方语义或确定性规则大量冲突。

#### H4：字段感知检索优于无知识和简单知识注入

假设：

> 在固定模型、Prompt、上下文预算和输出格式时，字段感知混合检索优于无知识、整库 Prompt、Exact、BM25 和纯 Dense。

否证条件：

- 不优于等上下文预算的整库输入或 Exact；
- 检索指标提高，但端到端结果和引用质量均不改善；
- shuffled 或 wrong-rule retrieval 也获得相同结果。

#### H5：Verifier 降低无依据输出

假设：

> 引用与规则验证器能够降低不存在字段引用、未满足规则触发和结论与证据不一致。

否证条件：

- unsupported claim 和 invalid trigger 没有下降；
- 攻击召回明显下降；
- 验证器只改写文本，没有拦截实际错误。

#### H6：方法可以跨设备、工具、来源和时间泛化

否证条件：

- ID 测试有效，但 leave-one-tool、leave-one-family 或 leave-one-provider 后优势消失；
- 主要收益来自检索同设备案例；
- 时间外推后性能失效。
