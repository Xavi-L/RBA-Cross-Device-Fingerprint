讲稿：

各位老师好，我的毕业论文题目是《基于三端融合设备指纹的移动端无感风控系统设计与实现》。本文面向移动端 Hybrid App 场景，设计并实现了一个原型系统 HybridGuard。

我这篇论文的核心不是提出一种新的机器学习算法，而是解决一个系统问题：在同一个移动端会话中，Android Native、WebView 宿主和 Web 前端三类原本割裂的指纹源，能不能被采集、对齐，并进一步转化为可解释的风险判断依据。接下来我会从研究背景、系统设计、关键实现、实验结果和总结展望五个部分进行汇报。

## 第2页：研究背景与问题

预计时间：1分20秒

建议展示：

- 用三端关系图或文字图说明 Native、WebView、Web 三层的边界
- 可引用论文中的三端特征来源表

| 特征来源 | 主要采集内容 | 风控意义 |
|---|---|---|
| Android Native | 设备型号、系统版本、Build、屏幕、电池、传感器、ADB | 描述底层设备真实性和系统环境 |
| WebView宿主 | JSBridge、WebView provider、宿主UA、安装来源、SDK配置 | 判断页面是否运行在预期App容器内 |
| Web前端 | Navigator、逻辑屏幕、DPR、WebGL、Canvas、时区、算力 | 描述网页脚本实际看到的运行环境 |

讲稿：

移动端账号安全和业务风控中，一个很重要的问题是如何在不打扰正常用户的情况下识别异常环境。传统验证码、短信校验可以提高攻击成本，但会打断用户流程，所以风控系统更希望在后台利用设备、环境和上下文信息做无感判断。

在 Hybrid App 中，一个业务会话实际上同时经过三层环境：第一层是 Android Native，它能看到设备型号、系统版本、物理屏幕、电池和传感器；第二层是 WebView 宿主，它能反映 JSBridge、WebView provider、安装来源和调试配置；第三层是 Web 前端，它能看到 UA、DPR、WebGL、Canvas、时区和算力表现。

问题在于，单一来源的指纹容易被局部伪造或局部遮蔽。只看 Web，可能被无头浏览器或脚本伪造；只看 Native，又看不到页面实际运行环境；只看 WebView，也无法完整判断底层物理设备是否可信。因此本文关注的不是“多采几个字段”，而是这些字段在同一会话里能否相互印证、发现矛盾。

| 移动端无感认证 | 行为、传感器、上下文可支持低打扰认证 | 更关注设备环境、宿主容器和前端运行时的一致性 |

| 端侧推理 | 轻量模型可降低延迟和数据外传 | 将离线规则标注能力压缩为App本地评分闭环 |

讲稿：

从相关研究看，浏览器指纹、移动端无感认证、WebView安全和端侧推理都已经有较多基础。但这些方向各自关注的对象不完全一样。

浏览器指纹研究通常更关注 Web API 暴露出的字段；移动端无感认证更多关注传感器、触摸行为或上下文；WebView 安全研究重点是漏洞、信息流和 JSBridge 风险；端侧推理研究则关注模型如何部署到设备上。

而本文的切入点是 Hybrid App 中的“跨层一致性”。也就是说，我不是只把 Web 指纹、Native 指纹和 WebView 信息并排收集起来，而是把它们放在同一个 `session_id` 下，看这些层之间是否像来自同一台设备、同一个 App 宿主和同一次真实会话。这也是后面系统设计和实验消融围绕“三端语义关系”展开的原因。

## 第4页：研究目标与主要工作

预计时间：1分钟

建议展示：

- 用四点贡献列表
- 可配合图2-4跨层一致性分析图

![图2-4 跨层一致性分析](../thesis_materials/top_journal_figures/fig2-4_cross_layer_consistency_analysis.png)

讲稿：

围绕这个问题，本文的目标是设计一个面向 Hybrid App 的三端融合设备指纹无感风控系统。系统需要完成五件事。

第一，设计三端设备指纹采集模型，把 Native、WebView 和 Web 前端分层采集并统一到同一个 `session_id` 下。

第二，实现后端会话合并和持久化，支持三端异步上报，并保存原始 JSON 和训练用 JSONL。

第三，构建跨层语义规则知识库，用来表达屏幕一致性、UA 一致性、WebView 内核一致性、JSBridge 宿主真实性、传感器可信性等规则。

第四，基于规则知识库和本地大模型生成离线风险标签，再训练轻量评分器。

第五，实现 Android 端侧评分闭环，使 App 可以在本地完成采集、编码和风险分输出，只上传评分摘要。

## 第5页：系统总体架构

预计时间：1分30秒

建议展示：

![图2-1 系统总体架构](../thesis_materials/top_journal_figures/fig2-1_system_architecture.png)

讲稿：

这一页是系统总体架构。HybridGuard 可以分成四层。

最左侧是采集层，包含 Android Native、WebView 宿主和 Web 前端三类探针。Native 负责底层设备和物理环境，WebView 负责 App 宿主和桥接能力，Web 前端负责浏览器运行时和渲染环境。

中间是服务与数据层，后端使用 FastAPI 接收三端数据，并通过 `session_id` 合并同一会话。因为 Native 数据和 Web/WebView 数据不是同一时刻到达，所以这里采用增量合并，避免后到达的数据覆盖已有字段。

再往右是规则与训练层。系统把采集到的三端字段送入规则知识库和本地大模型，生成带解释的 `risk_score` 和 `risk_reason`，再训练随机森林等轻量评分器。

最右侧是端侧评分层。训练阶段确定的字段顺序、类别编码和缺失值处理方式，会固化到 Android 端；运行时 App 本地完成三端采集、特征编码和随机森林推理，最后只上传风险分、风险等级和摘要信息。

这里我想强调一点：这套架构包含两条链路，上半部分是离线标注训练链路，下半部分是端侧评分运行链路。两条链路通过统一特征定义和编码策略连接起来。

## 第6页：离线标注训练链路

预计时间：1分20秒

建议展示：

![图2-2 离线标注训练流程](../thesis_materials/top_journal_figures/fig2-2_offline_label_training_flow.png)

表3-2 跨层语义规则类型与风险解释

| 规则类别 | 关联特征层 | 规则含义 | 风险解释 |
|---|---|---|---|
| 屏幕一致性 | Native屏幕 + Web屏幕 | `逻辑分辨率x DPR`应近似对应物理分辨率 | 偏差过大可能说明Web环境被伪造或脱离宿主 |
| UA一致性 | Native Build + WebView UA + Web UA | 系统版本、机型和内核版本应相互呼应 | 不一致可能说明UA被改写或请求被重放 |
| 宿主真实性 | JSBridge + session_id | Web页面应能通过JSBridge获取同一会话 | JSBridge缺失可能说明外部浏览器或脚本绕过App |
| 物理可信性 | 传感器 + 电池 | 真机通常具备合理传感器矩阵和动态电池状态 | 传感器极少或电池死值可能说明模拟器 |
| 渲染环境 | WebGL + Native硬件 | GPU renderer应符合移动端硬件生态 | SwiftShader、Headless、PC GPU属于高风险线索 |

讲稿：

这一页展示的是离线标注训练链路。旧采集 App 启动后先生成 `session_id`，Native 层采集原生特征并上报后端；随后 WebView 加载探针页面，页面通过 JSBridge 获取宿主信息，同时由 Web 前端采集 UA、屏幕、WebGL、Canvas 和算力等特征。后端按同一个 `session_id` 合并后，形成完整三端样本。

接下来，系统利用规则知识库和本地大模型生成风险标签。这里的大模型不是在线风控依赖，而是离线辅助标注工具。它的作用是在规则知识库约束下，对每条样本输出 `risk_score` 和 `risk_reason`。比如屏幕是否匹配、Native 设备型号是否能在 Web UA 中得到印证、JSBridge 是否存在、WebGL 是否暴露 SwiftShader 或 Headless 特征等。

这样做的好处是：复杂的跨层语义分析可以在离线阶段完成，而在线端侧只需要运行轻量评分器。

## 第7页：端侧评分运行链路

预计时间：1分30秒

建议展示：

![图2-3 端侧评分运行流程](../thesis_materials/top_journal_figures/fig2-3_on_device_scoring_flow.png)

![图3-1 端侧评分模块结构](../thesis_materials/top_journal_figures/fig3-1_on_device_module_structure.png)

讲稿：

端侧评分链路运行在新的 `riskapp` 模块中。与离线采集链路不同，这个 App 不再依赖后端托管探针页面，而是加载本地 `assets/local_probe.html`。App 启动后生成会话标识，Native 层先在本机采集并展平特征；WebView 加载本地探针页面，Web 探针通过 `ScoringWebBridge` 把 WebView 和 Web payload 回传给 Kotlin 层。

采集完成后，`RiskFeatureEncoder.java` 会按照训练阶段固定的 65 维字段顺序构造 `double[] input`。这一步很关键，因为训练侧和端侧的字段顺序、类别编码、缺失值策略必须完全一致。之后 App 调用 m2cgen 导出的 `DeviceRiskScorer.java` 执行随机森林推理，得到 0 到 100 的风险分。

最后，App 只上传 `session_id`、`risk_score`、`risk_level`、`risk_reason`、`scoring_engine` 和 `feature_count` 等摘要字段，不上传完整原始三端指纹。这体现了本文的端侧闭环和数据最小化设计。

## 第9页：轻量评分器选型与端侧部署取舍

预计时间：1分钟

建议展示：

- 可引用论文大纲中的随机森林与浅层 MLP 对照结果
- 重点讲“工程选型”，不要包装成算法创新

| 模型 | 测试样本数 | MAE | 中位误差 | P90误差 | 最大误差 |
|---|---:|---:|---:|---:|---:|
| 随机森林 | 265 | 1.14 | 1.11 | 2.25 | 6.42 |
| 浅层MLP | 265 | 2.42 | 2.04 | 5.23 | 13.29 |

讲稿：

这一页主要解释为什么端侧评分器选择随机森林，而不是更复杂的神经网络。本文的评分器不是算法创新点，而是系统闭环中的工程组件，所以选择标准主要是稳定、轻量、容易部署和容易解释。

从对照结果看，在相同测试样本上，随机森林的 MAE 为 1.14，浅层 MLP 的 MAE 为 2.42；随机森林的 P90 误差和最大误差也更低。原因与数据形态有关：本文处理的是小样本、结构化、混合数值与类别字段的设备指纹数据，而不是图像、语音这类高维连续数据。

另外，随机森林可以通过 m2cgen 导出为 Java 代码，直接集成到 Android 的 `DeviceRiskScorer.java` 中；同时树模型的特征重要性也方便解释风险来源。因此，本文最终把随机森林作为端侧评分器的主要实现，把 MLP 作为工程对照。

## 第11页：为什么需要分组交叉验证

预计时间：1分20秒

建议展示：

![图4实验 分组交叉验证各折构成](../ablation/figures/figure_02_fold_distribution.png)

![图4实验 随机留出与分组交叉验证误差对比](../ablation/figures/figure_03_holdout_vs_grouped_mae.png)

表4-4 分组交叉验证各折样本分布

| Fold | `physical_device` | `cloud_device` | `script_attack` |
|---:|---|---|---|
| 1 | 82 rows / 21 groups | 290 rows / 23 groups | 104 rows / 1 group |
| 2 | 128 rows / 21 groups | 236 rows / 28 groups | 97 rows / 1 group |
| 3 | 76 rows / 23 groups | 211 rows / 26 groups | 99 rows / 1 group |

讲稿：

除了普通的随机 holdout，本文还设计了分组交叉验证。原因是当前数据里存在真实样本扩充、云真机重复采集和脚本模板生成。如果只按行随机切分，同一设备的相似样本或者同一攻击模板的相似样本可能同时进入训练集和测试集，导致指标偏乐观。

分组交叉验证按 `group_id` 切分，保证同一真实设备、同一云测设备或同一攻击模板不会同时出现在训练和测试中。由于脚本攻击样本有三个主模板，所以本文采用三折分组验证，并保证每一折都包含真实物理设备、云测设备和一个未见脚本攻击模板。

这张 holdout 与 grouped MAE 对比图说明，分组后部分配置误差明显上升，尤其是只使用 Web 特征的配置。这说明分组验证确实更严格，也更接近真实部署时面对未见设备和未见攻击变体的情况。

同时也可以看到，`Consistency only` 全部 38 个一致性特征在分组验证下不如 `Raw all`，说明并不是所有一致性特征都稳定。真正有效的是经过筛选和抽象的核心三端语义规则。这也是本文实验结论比较保守的地方：不是说所有规则都泛化，而是说明核心跨层语义规则具有更好的稳定性。

## 第14页：特征重要性与可解释性

预计时间：1分20秒

建议展示：

![图4实验 跨层一致性特征重要性](../ablation/figures/figure_05_consistency_feature_importance.png)

表4-13 重要一致性特征及风险解释

| 特征 | 重要性 | 解释 |
|---|---:|---|
| `consistency_tri_layer_sensor_bridge_fail` | 0.1518 | 传感器严重缺失或JSBridge未注入，对高风险样本关键 |
| `consistency_native_webview_debug_cleartext_tension` | 0.1017 | debug与明文网络配置共同反映测试或调试宿主特征 |
| `consistency_native_web_model_ua_strength` | 0.0850 | Native设备身份与Web UA是否能相互印证 |
| `consistency_tri_layer_failure_count` | 0.0844 | 多个一致性检查失败的聚合信号 |
| `consistency_webview_web_ua_has_wv_token` | 0.0843 | Web UA是否呈现WebView运行时特征 |
| `consistency_native_web_model_ua_match` | 0.0839 | Native型号是否与Web UA直接匹配 |
| `consistency_tri_layer_manual_timezone_or_adb` | 0.0749 | manual安装来源结合时区或ADB的云机房风险 |

讲稿：

最后看一下特征重要性。这里使用的是 `Consistency only` 配置下随机森林的特征重要性，用来解释模型到底学到了什么。

排名第一的是 `consistency_tri_layer_sensor_bridge_fail`，也就是传感器严重缺失或 JSBridge 未注入。这个特征同时涉及 Native 的物理可信性和 WebView 的宿主真实性，对高风险样本非常关键。

第二类重要特征是 debug、cleartext、manual 安装来源、ADB 和时区等组合信号。这些字段单独出现不一定高危，但组合出现时更像测试机架、云真机或群控环境。

第三类是 Native 设备身份和 Web UA 的匹配，例如型号、品牌、Android 版本是否能在 Web UA 中得到印证。第四类是 WebView-Web 之间的运行时标记，例如 Web UA 是否包含 WebView 的 `wv` token。

这些结果说明，模型不是单纯记忆原始字符串，而是在利用已经编码好的跨层语义关系。这也对应本文系统贡献：HybridGuard 的核心是建立 Native、WebView 和 Web 前端之间可采集、可对齐、可解释的设备指纹关系。

预计时间：1分钟

建议展示：

讲稿：

总结一下，本文完成了一个面向 Hybrid App 的三端融合设备指纹无感风控原型系统 HybridGuard。

第一，系统实现了 Android Native、WebView 宿主和 Web 前端三端指纹采集，并通过 `session_id` 完成会话对齐和后端持久化。

第二，本文构建了跨层语义规则知识库，并利用规则知识库驱动本地大模型生成离线风险标签，再训练端侧轻量评分器。

第三，本文实现了 Android 端侧评分闭环，App 在本地完成三端采集、65 维特征编码和随机森林推理，并只上传评分摘要。

实验结果表明，三端原始字段之间存在冗余，但这种冗余恰好是跨层互证的基础。尤其在分组交叉验证下，7 个三端语义规则特征取得了优于完整原始三端特征的 MAE 和 RMSE，说明跨层一致性建模具有一定泛化能力。

当然，本文仍有不足：真实设备和真实攻击样本规模有限，风险标签还需要更多人工复核，端侧运行开销也需要在更多真机上测试。后续可以继续扩展真实设备覆盖，构造更多跨层矛盾专项样本，并进一步研究规则执行引擎或模型压缩方案。

## 可选备答口径

### 1. 为什么完整三端原始特征不是所有实验里最优？

可以回答：完整三端字段不是简单越多越好。Native 设备型号、Web UA、WebView provider、屏幕和 DPR 之间存在天然代理关系，直接拼接可能带来冗余甚至轻微噪声。本文的核心结论不是“字段越多越强”，而是“跨层关系更重要”。分组验证中 `Tri-layer semantic` 仅 7 个特征优于 65 维 Raw all，正好说明了这一点。

### 2. 为什么选择随机森林，而不是深度学习模型？

可以回答：本文不是算法创新论文，随机森林主要是端侧轻量评分器。三端指纹是小样本、结构化、混合数值和类别的数据，随机森林更稳定、训练成本低、可解释性更好，并且可以通过 m2cgen 导出 Java 代码集成到 Android 端。MLP 在本文中是工程对照，不是主贡献。

### 3. 大模型标签是否可靠？

可以回答：本文没有让大模型自由打分，而是用结构化规则知识库约束大模型输出。每条样本输出 `risk_score` 和 `risk_reason`，风险解释可以回溯到规则类别。当前标签仍然需要更多人工抽样复核，所以论文中也明确把它作为原型阶段的离线规则标签，而不是最终业务人工标签。

### 4. 数据集规模有限会不会影响结论？

可以回答：会影响结论外推范围，所以本文没有声称覆盖所有真实业务攻击环境。实验重点是验证原型系统中三端采集、会话对齐和跨层一致性建模是否有效。为降低同源样本影响，本文额外采用分组交叉验证，让同一设备或同一攻击模板不跨训练测试划分。

### 5. 端侧只上传摘要的意义是什么？

可以回答：设备指纹具有隐私敏感性。端侧评分链路在 App 本地完成采集、编码和模型推理，服务端只接收风险分、风险等级和摘要字段，减少原始三端指纹长期外传的必要性。这对应本文的数据最小化和端侧闭环设计目标。
