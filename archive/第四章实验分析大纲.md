## 4.2 实验目标、实验设计与评估指标

### 4.2.1 实验目标

建议正文内容：

本章围绕三个问题展开：

1. Android Native、WebView 宿主、Web 前端三端原始特征分别包含多少风险信息？
2. 三端之间的跨层一致性关系是否比简单拼接原始字段更能体现系统价值？
3. 在避免同源设备或同源攻击模板泄漏后，跨层一致性特征是否仍具有泛化能力？

写作重点：

- 把实验目标和论文贡献绑定起来：本文贡献不是新模型，而是三端融合采集与跨层语义对齐。
- 说明消融实验正好服务于“证明系统设计必要性”。

建议篇幅：400-600 字。

### 4.2.2 评估指标

建议正文内容：

1. 回归指标：

```text
MAE = 平均绝对误差
RMSE = 均方根误差
R2 = 决定系数
```

2. 辅助分类指标：

```text
Precision、Recall、F1、Accuracy
高风险阈值：risk_score >= 80
```

3. 解释为什么主看 MAE/RMSE：

- 风险评分是连续值，系统不仅要区分高低风险，也要尽量贴近标签分数。
- 高风险 F1 在随机划分下容易饱和，不能单独作为主要结论。
- RMSE 对大误差更敏感，可辅助观察未见设备或未见模板上的泛化风险。

建议篇幅：500-700 字。

### 4.3.3 随机 Holdout 结果与分析

建议正文内容：

| 配置 | 特征数 | MAE | RMSE | R2 | 高风险 F1 |
| --- | ---: | ---: | ---: | ---: | ---: |
| Native only | 33 | 1.222 | 2.198 | 0.9933 | 1.000 |
| WebView only | 14 | 1.139 | 1.537 | 0.9967 | 1.000 |
| Web only | 18 | 1.416 | 2.377 | 0.9922 | 1.000 |
| Native + WebView | 47 | 1.129 | 1.515 | 0.9968 | 1.000 |
| Native + Web | 51 | 1.219 | 2.177 | 0.9934 | 1.000 |
| WebView + Web | 32 | 1.131 | 1.537 | 0.9967 | 1.000 |
| Native + WebView + Web | 65 | 1.140 | 1.544 | 0.9967 | 1.000 |

分析口径：

1. 所有配置在随机 holdout 下误差都较低，说明当前风险标签中存在强规则信号。
2. 完整三端并不是 MAE/RMSE 的绝对最优，这不是实验失败。
3. 原因是三端原始字段存在大量代理关系：
   - Native 设备型号、系统版本和屏幕信息会反映到 Web UA、DPR 和逻辑屏幕。
   - WebView Provider 版本会反映到 Web UA 的 Chrome/WebView 版本。
   - WebView 层的 JSBridge、安装来源、debuggable 等字段与风险标签强相关。
4. 因此，按端粗粒度删除不能充分体现三端融合的核心价值。

建议篇幅：900-1200 字。

| --- | ---: | ---: | ---: | ---: | ---: |
| Native only | 33 | 2.799 | 1.541 | 4.774 | 0.667 |
| WebView only | 14 | 1.541 | 0.435 | 2.643 | 1.000 |

| WebView + Web | 32 | 3.053 | 0.994 | 5.100 | 0.978 |
| Native + WebView + Web | 65 | 2.642 | 1.004 | 4.455 | 1.000 |

分析口径：

1. 分组后整体误差上升，说明随机 holdout 存在同源样本相似性带来的乐观估计。
2. Web only 在分组验证中误差显著升高，说明单独 Web 前端特征对未见设备或未见模板泛化不足。
3. WebView only 表现较强，主要因为当前数据中 JSBridge、安装来源、debuggable、Provider 等字段与规则标签高度相关。
4. Native + WebView 组合优于完整三端原始特征，说明原始字段拼接可能引入冗余或噪声。
5. 该实验进一步引出下一节：与其堆叠更多原始字段，不如显式建模跨层一致性关系。

## 4.4 跨层一致性特征构造与消融实验

### 4.4.1 实验目的

建议正文内容：

粗粒度消融只能回答“哪一端字段有信息量”，但不能直接回答“Native、WebView、Web 三端之间是否形成了可互证的语义关系”。因此，本节将三端之间的语义对齐关系编码为显式一致性特征，再评估这些特征对风险分拟合的作用。

核心论点：

```text
三端融合的价值不在于字段数量增加，而在于不同层级之间可以相互校验、相互印证、发现矛盾。
```

建议篇幅：400-600 字。

### 4.4.2 一致性特征构造方法

建议正文内容：

说明一致性特征来自 `ablation/run_consistency_ablation.py` 和 `ablation/consistency_feature_rules.md`，共 38 个，分为四组：

| 分组 | 特征数量 | 语义 |
| --- | ---: | --- |
| Native-Web | 18 | Android 底层环境与 Web JS 暴露环境是否一致 |
| Native-WebView | 8 | Android 底层环境与 App/WebView 宿主是否一致 |
| WebView-Web | 5 | WebView 容器与 Web 前端运行环境是否一致 |
| Tri-layer semantic | 7 | 三端共同参与的核心风险规则与综合一致性分 |

代表性特征：

| 特征 | 含义 |
| --- | --- |
| `consistency_native_web_model_ua_strength` | Native 设备型号、产品、主板、品牌在 Web UA 中的宽松匹配强度 |
| `consistency_native_web_android_version_delta` | Native Android 版本与 Web UA Android 版本差异 |
| `consistency_native_web_screen_score` | 物理屏幕、Web 逻辑屏幕、DPR 的软一致性分 |
| `consistency_native_web_gpu_family_score` | Native 硬件族与 WebGL GPU 族的宽松匹配分 |
| `consistency_native_webview_agent_model_match` | Native 设备型号是否出现在 WebView system HTTP agent |
| `consistency_webview_web_chrome_major_match` | WebView Provider 与 Web UA Chrome 主版本是否一致 |
| `consistency_tri_layer_sensor_bridge_fail` | 传感器严重缺失或 JSBridge 未注入 |
| `consistency_tri_layer_failure_count` | 多个一致性检查失败的聚合计数 |

写作重点：

- 重点解释“宽松匹配”，不是字符串完全相等。
- 例子：`qcom/Qualcomm` 与 `Adreno` 属于语义一致；物理分辨率和 Web 逻辑分辨率要乘 DPR，并允许状态栏或导航栏误差。

建议篇幅：1000-1300 字。

| 配置 | 含义 |

建议篇幅：700-900 字。

### 4.5.3 结果分析

建议正文分析逻辑：

1. 分组验证下整体误差高于随机 holdout，说明分组评估更严格。
2. `Consistency only` 不再优于 `Raw all`，说明当测试集包含未见模板时，并非所有一致性特征都稳定泛化。
3. 在一致性相关配置中，`Tri-layer semantic` 取得最低 MAE 和 RMSE，说明少量核心三端语义规则比完整原始字段更稳。
4. `Native-WebView consistency` 表现较好，说明当前数据集中 App 宿主真实性、JSBridge、安装来源和调试配置是重要风险信号。
5. `Native-Web consistency` 在分组验证中误差较大，可能是因为不同设备的 UA、屏幕、GPU、WebGL 组合差异较大，单独使用这组特征不足以处理未见设备。
6. `Raw all + Consistency` 没有提升，说明简单增加特征数量不一定提升泛化能力。

核心结论建议写成：

> 在更严格的分组交叉验证下，三端语义规则特征仅使用 7 个特征便取得优于完整三端原始特征的误差表现，说明跨层一致性建模不是依赖同源样本记忆，而是在未见设备和未见攻击模板上仍具有一定泛化能力。

建议篇幅：1200-1500 字。

前几节主要从误差指标上说明一致性特征有效。本节进一步分析随机森林在 `Consistency only` 配置下依赖哪些特征，以验证模型是否确实利用了跨层语义关系，而不是只记忆原始字符串或单端字段。

### 4.6.3 规则解释与系统贡献对应关系

建议正文内容：

按规则类别展开解释：

1. 宿主真实性：JSBridge 注入、WebView UA 中的 `wv` 标记、WebView Provider 与 Chrome 主版本一致性。
2. 设备身份一致性：Native 型号、产品、品牌与 Web UA 的宽松匹配。
3. 物理可信性：传感器矩阵、ADB、电池状态、安装来源等组合信号。
4. 渲染环境一致性：Native 硬件族与 WebGL GPU 族的软匹配，SwiftShader 或 Headless 痕迹。
5. 聚合异常：多个一致性检查同时失败时，风险分显著升高。

- 把特征重要性和第 2、3 章系统设计对应起来。

## 4.7 本章讨论与局限性

### 4.7.1 实验结论汇总

建议正文内容：

总结四个结论：

1. 粗粒度三端消融显示，单端或双端原始字段在随机划分下也能取得较低误差，说明三端原始字段之间存在较强冗余。
2. 跨层一致性消融显示，显式一致性特征可以达到与完整原始三端特征相近甚至更优的效果，说明跨层语义比对本身是有效风险信号。
3. 分组交叉验证显示，随机 holdout 会高估泛化能力；在更严格分组设置下，`Tri-layer semantic` 仍优于 `Raw all`。
4. 特征重要性分析显示，模型重点依赖传感器与 JSBridge 完整性、UA 一致性、WebView 宿主特征、manual 安装来源与 ADB 等跨层语义信号。

建议篇幅：700-900 字。

### 4.7.2 局限性说明

建议正文内容：

建议坦诚写以下局限：

1. 当前数据集规模仍有限，真实物理设备和真实攻击环境覆盖不足。
2. 高风险攻击样本主要来自模板化构造，后续需要更多真实攻击链路样本。
3. 风险标签由规则知识库驱动的大模型离线标注流程生成，虽然具备可解释性，但仍需要人工复核或业务标注进一步验证；综合规则知识库可作为后续标签复核和规则迭代的基础。
4. 本章主要是离线消融实验，没有把端侧运行耗时、功耗、内存和大规模真机闭环作为主实验。
5. 当前分组策略依赖启发式 `source_type` 和 `group_id` 构造，后续采集阶段可直接记录设备组和攻击模板 ID，提高评估严谨性。

写作重点：

- 局限不要写得像“实验失败”，而要写成“后续改进方向”。
- 强调这些局限不影响本章对“三端跨层一致性有效性”的核心验证。

建议篇幅：700-900 字。

### 4.7.3 后续改进方向

建议正文内容：

1. 扩大真实设备覆盖，增加不同品牌、系统版本、WebView 内核版本和网络环境。
2. 构造更多跨层矛盾专项样本，例如 Android 真机 + Windows UA、屏幕 DPR 不一致、WebGL 软件渲染、JSBridge 缺失等。
3. 引入人工抽样复核，提升风险标签可信度。
4. 对端侧评分 App 进行系统开销测试，包括采集耗时、推理耗时、内存占用和模型代码体积。
5. 研究更轻量的规则执行或模型压缩方案，使三端语义规则更适合移动端部署。

建议篇幅：500-700 字。

## 4.8 本章小结

建议正文内容：

本章基于已标注的三端设备指纹数据，围绕三端融合有效性和跨层一致性建模展开实验分析。首先介绍了数据来源、标签分布、采集字段完整性和实验评估方式；随后通过粗粒度三端消融说明 Native、WebView、Web 三类原始字段均包含风险信息，但原始字段之间存在较强冗余，简单拼接并不能完全体现三端融合价值；进一步通过一致性特征消融说明，将设备型号与 UA、Android 版本、屏幕 DPR、WebView Provider、JSBridge、传感器、安装来源等跨层关系显式编码后，可以得到有效风险信号；最后通过分组交叉验证证明，在避免同源设备和同源攻击模板泄漏后，三端核心语义规则仍具有较好的泛化能力。实验结果表明，本文系统的关键价值不是提出新的机器学习模型，而是建立 Android Native、WebView 宿主和 Web 前端之间可采集、可对齐、可解释的跨层设备指纹关系。

## 可直接写入正文的核心结论句

1. 粗粒度三端消融结果表明，完整三端原始特征并未在所有指标上显著优于双端组合，这说明三端原始字段之间存在较强冗余和代理关系。
