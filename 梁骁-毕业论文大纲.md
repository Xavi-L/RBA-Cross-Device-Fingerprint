## 前置部分

### 题目

基于三端融合设备指纹的移动端无感风控系统设计与实现

### 摘要

围绕移动端账号安全、反自动化攻击和风险自适应认证场景，说明单一 Web 指纹或单一 Native 指纹容易受到模拟器、无头浏览器、接口重放和云真机环境干扰。本文重点不是提出新的机器学习算法，而是设计并实现一种面向 Hybrid App 场景的三端融合设备指纹系统。系统通过 Android Native、WebView 宿主容器和 Web 前端运行环境三类数据源采集多层设备特征，在同一会话下完成异步数据对齐、跨层语义规则构建、一致性校验、风险标签生成和端侧轻量评分。本文利用大语言模型分析不同层级特征之间的语义对齐关系，形成规则知识库，使风险判断不局限于固定阈值，而能够基于跨层特征之间的相互呼应进行综合评分。实验部分重点评估三端采集完整性、跨端融合有效性、规则知识库有效性、系统开销和本地评分闭环可行性；随机森林模型仅作为系统闭环中的轻量评分实现，用于验证系统采集到的三端特征可以支持风险判断。

## 1 绪论

### 1.1 研究背景与意义

1. 移动互联网账号安全与无感风控需求增长。
2. 传统验证码、短信验证等显式认证方式对用户体验的影响。
3. 设备指纹在风险自适应认证中的作用：辅助识别可信设备、异常环境和自动化攻击。
4. 单端指纹的局限：浏览器指纹易被脚本伪造，Native 指纹缺少前端渲染与行为环境信息，WebView 容器被剥离后容易出现接口重放。
5. 三端融合指纹的研究意义：通过 Android Native、WebView 和 Web 三端特征交叉验证，提高模拟器、无头浏览器、云真机和 API 重放攻击识别能力。本文的核心贡献在于系统性地打通这三类原本割裂的数据源，并建立同一会话下的采集、对齐和验证链路。

### 1.2 国内外研究现状

#### 1.2.1 设备指纹与浏览器指纹研究

介绍基于 User-Agent、屏幕、Canvas、WebGL、硬件并发数、时区等 Web 指纹特征的研究现状，以及指纹稳定性、唯一性和隐私约束。

#### 1.2.2 移动端风控与模拟器检测研究

介绍移动端设备认证、模拟器检测、Root/ADB 状态、传感器矩阵、电池状态、系统构建指纹等 Native 特征在风控中的应用。

#### 1.2.3 Hybrid App 宿主环境与端侧风控系统研究

介绍 Hybrid App 中 WebView 宿主环境、JSBridge、Native 与 Web 边界、端侧采集和本地风控闭环相关研究，突出已有工作较少把 Native、WebView 宿主和 Web 前端作为一个整体系统进行融合验证。

#### 1.2.4 相关研究总结

总结现有方法的不足：多依赖单端特征、服务端判定链路较重、对 Hybrid App 中 WebView 宿主一致性关注不足、缺少面向端侧闭环评分的系统实现。引出本文的三端融合设备指纹系统方案。注意本节不展开复杂算法比较，只说明本文选择轻量评分模型是为了服务系统验证和端侧部署。

### 1.3 研究目标与研究内容

1. 设计三端设备指纹采集模型：Android Native、WebView 宿主容器、Web 前端运行环境。
2. 实现 HybridGuard 原型 App 与 FastAPI 后端，完成同一会话下多源数据合并与持久化。
3. 构建跨层语义规则知识库：利用大语言模型分析不同层级特征之间的语义对齐关系，沉淀屏幕一致性、UA 一致性、宿主容器存在性、硬件物理可信性、渲染与算力一致性等规则。
4. 构建风险评分数据集：采集真实设备数据、扩充真实样本、生成高危攻击样本，并基于规则知识库和本地大模型生成风险标签。
5. 建立跨端一致性校验和风险标签生成流程，使采集系统能够输出可用于评估的风险分数。
6. 完成 App 本地评分闭环：将轻量评分模块集成进 Android App，实现采集数据后本地输出风险分数。
7. 设计并完成三端特征消融实验，验证 Native、WebView、Web 三类系统特征源的必要性。

说明全文结构：第 2 章系统设计与总体架构，第 3 章关键模块设计与实现，第 4 章实验分析，最后给出结论、致谢和参考文献。

## 2 系统设计与总体架构

### 2.1 系统设计目标

1. 三端融合一致性：同一设备在 Native、WebView、Web 三端特征之间应具备可解释的一致关系。
2. 无感采集：尽量在用户无额外交互的情况下完成风险特征采集。
3. 语义规则化：系统能够把不同层级特征之间的语义呼应关系沉淀为规则知识库，避免只使用单一硬阈值进行机械判断。
4. 本地闭环：系统能够在 Android 端完成采集和轻量评分，减少对远程风控服务的依赖。
5. 可扩展性：后续可继续增加行为特征、网络特征或替换评分器，而不改变三端采集架构。
6. 实验可复现：数据采集、规则构建、风险标注、模型训练和评估过程可被独立执行。

### 2.2 功能需求分析

#### 2.2.1 跨端设备指纹采集功能

Android Native 侧采集系统构建、内存、物理屏幕、电池动态、传感器矩阵、安全配置等特征；WebView 侧采集 JSBridge、WebView 内核、宿主安全、安装时间和 SDK 信息；Web 侧采集 Navigator、逻辑屏幕、WebGL、Canvas、算力挑战和时区等特征。

#### 2.2.2 会话合并与数据持久化功能

以 `session_id` 为主键合并 Native 与 Web/WebView 上报数据，保留原始嵌套结构，同时生成面向训练和标注的扁平化 JSONL 数据。

#### 2.2.3 规则知识库与风险标签生成功能

利用本地大模型分析不同层级 feature 的语义对齐关系，形成规则知识库；再基于这些规则对设备指纹进行综合风险评分，生成 `risk_score` 与 `risk_reason`。规则知识库用于表达“跨层设备指纹可以相互形成呼应”的系统思想，而不是依赖单个字段的死板阈值。

#### 2.2.4 轻量评分与 App 本地评分功能

由于大模型在线调用存在成本、延迟、隐私和离线可用性问题，需要将规则知识库产生的评分能力压缩为端侧轻量评分器。将训练生成的 `DeviceRiskScorer.java` 与特征向量构造逻辑集成进 Android App，使 App 能够在完成采集后直接输出本地风险分数。该部分为后续需要完成的核心系统工作。

#### 2.2.5 实验分析功能

支持采集完整性评估、系统开销评估、攻击样本案例分析、三端特征消融实验，以及轻量评分误差评估。

### 2.3 非功能需求分析

#### 2.3.1 识别准确性与鲁棒性

系统应能区分正常真机、云真机/测试机架、廉价模拟器、无头浏览器和 API 重放等环境。

#### 2.3.2 端侧性能与资源占用

端侧评分模型需要控制推理耗时、代码体积和内存占用，避免影响 App 启动和 WebView 加载。

#### 2.3.3 隐私与数据最小化

论文中说明采集特征主要用于设备环境一致性验证，不直接采集用户身份内容、通讯录、文件等敏感个人数据。

#### 2.3.4 可维护性与可扩展性

采集字段、后端 Pydantic 模型、训练脚本和端侧评分模块应保持字段定义一致，便于后续添加新特征。

### 2.4 系统总体架构设计

系统分为四层：

1. 采集层：Android App、WebView JSBridge、前端探针页面。
2. 服务层：FastAPI 接口、会话合并、JSON/JSONL 持久化。
3. 规则与评分层：跨层语义规则知识库、真实数据扩充、高危样本生成、本地大模型评分、轻量评分器训练。
4. 端侧决策层：Android 本地特征向量构造、轻量风险评分、风险结果展示或上报。

### 2.5 系统工作流程

1. App 启动后生成 `session_id`。
2. Native 线程采集 Android 原生特征并上报。
3. WebView 加载前端探针页面，通过 JSBridge 读取宿主容器信息。
4. Web 前端采集浏览器运行环境、WebGL、Canvas 和算力挑战特征。
5. 后端按 `session_id` 合并多端数据并持久化。
6. 本地大模型根据跨层语义规则知识库生成风险标签。
7. 训练脚本将标注数据转化为评分器输入，训练轻量风险评分模型。
8. 评分器导出 Java 推理代码，最终集成进 App 完成本地评分闭环。

### 2.6 本章小结

概括系统目标、功能边界、总体架构和端到端数据流，为后续模块实现章节铺垫。

### 3.3 跨层语义规则知识库与风险标签生成模块

#### 3.3.1 规则知识库构建思路

利用大语言模型分析不同层 feature 的语义对齐关系，将三端特征之间的对应关系转化为可复用的规则知识。例如 Native 物理分辨率与 Web 逻辑分辨率/DPR 的对齐、Native Build 信息与 Web UA/System UA 的对齐、WebView JSBridge 与 App 宿主真实性的对齐、传感器/电池与真机物理属性的对齐、WebGL/Canvas/算力与移动端渲染环境的对齐。

#### 3.3.2 大模型辅助风险评分流程

介绍 `rba_engine.py` 与 `sorting.py` 调用本地 LM Studio 模型，对设备指纹 JSON 输出 `risk_score` 和 `risk_reason`。

#### 3.3.3 风险规则与提示词设计

概括提示词中的核心检测逻辑：传感器严重缺失、JSBridge 缺失、模拟器 GPU、PC/Headless UA、ADB 与高电量组合、安装来源异常、时区异常、跨层参数撕裂。强调规则不是简单 if-else 阈值表，而是通过 LLM 对跨层语义关系进行“软对齐”和综合解释，允许合理误差和设备差异存在。

#### 3.3.4 规则知识库示例

建议在论文中列出规则表：

| 规则类别 | 关联特征层 | 规则含义 | 风险解释 |
| --- | --- | --- | --- |
| 屏幕一致性 | Native 屏幕层 + Web 屏幕层 | 逻辑分辨率 x DPR 应近似对应物理分辨率 | 不一致可能说明 Web 环境被伪造或运行在非宿主容器中 |
| UA 一致性 | Native 构建层 + WebView 内核层 + Web Navigator | 系统版本、机型、WebView 版本应相互呼应 | 不一致可能说明 UA 被伪造或请求被重放 |
| 宿主真实性 | WebView Bridge 层 + Host 安全层 | JSBridge 应存在且能返回同一 `session_id` | 缺失可能说明脱离 App 的外部浏览器或脚本重放 |
| 物理可信性 | Native 传感器层 + 电池层 | 真机应具备合理传感器矩阵和动态电池状态 | 异常可能说明模拟器或虚拟环境 |
| 渲染环境一致性 | Web 图形层 + Native 硬件层 | GPU/WebGL Renderer 应符合移动端硬件生态 | SwiftShader、Headless、PC GPU 暴露高风险 |

### 3.5 轻量风险评分模块

#### 3.5.1 特征工程

说明将嵌套 JSON 展平；删除 `session_id`、`timestamp`、`client_ip`、大模型标签等非输入字段；对类别特征、布尔特征、数值特征分别进行编码、填补和标准化。

#### 3.5.2 为什么需要轻量评分器

说明系统不能在 App 运行时直接依赖大模型在线评分：一是大模型推理成本高，二是响应延迟不适合端侧实时风控，三是网络不可用时无法完成判断，四是原始指纹持续上传会带来隐私与工程部署压力。因此需要把规则知识库和大模型标注得到的判断能力，压缩为可在 Android 本地快速执行的轻量评分器。

#### 3.5.3 评分器候选实现

简要介绍 `train_mlp.py` 和 `train_randomforest.py` 两种候选评分器实现。写作时控制篇幅，只说明它们用于把三端系统采集结果转化为可部署风险分数，不将模型结构作为本文主要贡献。

#### 3.5.4 为什么选择随机森林而不是神经网络

从工程角度说明随机森林优于浅层神经网络：训练和调参成本更低；对小样本、混合类别/数值特征更稳定；不强依赖复杂标准化和高维独热输入；推理响应速度可控；可以通过树路径、特征重要性和分裂条件解释部分打分依据，更符合风控系统需要说明原因的场景。神经网络作为对照模型即可，不作为最终端侧实现。

#### 3.5.5 模型选择依据

当前测试集中随机森林 MAE 约 1.14，浅层 MLP MAE 约 2.42。该结果只用于支撑工程选型：本文选择误差更低、部署更直接的随机森林作为系统闭环评分器，而不声称算法层面的创新。

App 完成 Native、WebView、Web 三端采集后，在本地调用随机森林推理代码，输出风险分数和风险等级；必要时再上报后端用于审计。

#### 3.6.4 工程风险

重点处理训练特征与端侧特征顺序不一致、类别编码表缺失、Java 代码体积较大、WebView 异步采集未完成就触发评分等问题。

### 3.7 本章小结

总结三端采集、后端合并、规则知识库、大模型标注、数据集构建、轻量评分和端侧部署模块的实现关系。强调本章核心是系统链路实现，而不是新算法设计。

## 4 实验分析

### 4.1 实验环境配置

1. Android 客户端：HybridGuard App，minSdk 30，targetSdk/compileSdk 36，OkHttp 上报，WebView 加载前端探针。
2. 后端环境：FastAPI 数据接收服务，本地 JSON/JSONL 持久化。
3. 标注环境：LM Studio 本地大模型服务。
4. 评分器训练环境：Python、scikit-learn、PyTorch、m2cgen。
5. 数据采集环境：本地 Android 真机与 BrowserStack 云真机。

### 4.2 数据集说明

#### 4.2.1 数据来源

真实样本来自 App 在多台 Android 设备上的采集；扩充样本来自真实样本动态特征扰动；高危样本来自攻击脚本模拟。

#### 4.2.2 标签分布

当前 `training/scored_data.jsonl` 中风险分数覆盖 15、20、35、38、40、42、45、92、95、98 等区间，可划分为低风险、中风险和高风险三类进行进一步分析。

#### 4.2.3 数据预处理

说明缺失值填补、类别编码、标准化、训练/测试集划分，以及避免数据泄露的处理方式。

### 4.3 实验目标与评估指标

#### 4.3.1 三端采集完整性

统计 Native、WebView、Web 三端字段采集成功率、会话合并成功率、关键字段缺失率，验证系统能稳定形成完整的三端指纹。

#### 4.3.2 三端融合与跨层呼应有效性

通过消融实验比较完整三端特征与单端/双端特征的风险识别差异，验证本文系统架构的必要性；通过典型样本分析说明不同层设备指纹可以相互形成呼应。

#### 4.3.3 规则知识库有效性

评估 LLM 生成的跨层规则是否能覆盖主要风险场景，是否能对正常真机、云真机、模拟器、无头浏览器和 API 重放给出合理解释。

#### 4.3.4 端侧性能

评估本地评分耗时、App 启动/采集耗时、WebView 探针加载耗时、模型代码体积和内存占用。

#### 4.3.5 轻量评分效果

使用 MAE、Median Error、P90 Error、真实分与预测分散点图评估端侧评分器对大模型标签的拟合能力。该指标服务于系统闭环验证，不作为算法优劣的核心论证。

### 4.4 三端采集与会话合并实验

#### 4.4.1 实验设置

在本地真机和云真机环境中运行 App，分别统计 Native 上报、WebView/Web 上报、后端按 `session_id` 合并的成功情况。

#### 4.4.2 评估内容

建议统计三端字段数量、关键字段采集成功率、同一会话双路上报时间差、合并记录数量、异常会话数量。

#### 4.4.3 结果分析

分析异步上报、WebView 加载、网络环境和 JSBridge 注入对采集完整性的影响，突出系统工程实现的可靠性。

### 4.5 规则知识库与跨层对齐案例分析

#### 4.5.1 实验目的

验证 LLM 辅助构建的规则知识库能否把三端 feature 的语义关系解释清楚，并用于风险打分。

#### 4.5.2 案例设计

选取正常真机、云真机/测试机架、API 重放、无头 PC 浏览器、廉价模拟器等样本，展示其 Native、WebView、Web 三端特征如何相互印证或相互矛盾。

#### 4.5.3 结果分析

重点说明 LLM 的作用是进行不那么硬性/死板的跨层特征对齐：例如屏幕尺寸允许系统导航栏导致的轻微误差，ADB 在开发阶段不一定直接高危，Android 16/API 36 不应被误判为伪造系统。该实验服务于证明规则知识库比单一阈值表更适合处理真实设备差异。

### 4.6 轻量评分器工程选型实验

#### 4.6.1 实验设置

使用相同训练/测试划分，对随机森林与浅层 MLP 进行对比。

#### 4.6.2 为什么需要轻量模型

对比“大模型在线评分”和“端侧轻量评分”的成本、响应速度、离线可用性和隐私开销，说明轻量评分器是系统闭环必要组件，而不是算法创新点。

#### 4.6.3 为什么选择随机森林

从成本、响应速度、打分依据可解释性三方面说明选择随机森林而不是神经网络。当前仓库测试结果显示：

| 模型 | 测试样本数 | MAE | 中位误差 | P90 误差 | 最大误差 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 随机森林 | 265 | 1.14 | 1.11 | 2.25 | 6.42 |
| 浅层 MLP | 265 | 2.42 | 2.04 | 5.23 | 13.29 |

#### 4.6.4 结果分析

分析随机森林在当前系统数据上的误差更低、Java 导出更直接，因此适合作为端侧闭环评分器。答辩时只强调“工程选型”，不将其包装为算法创新。

### 4.7 三端特征消融实验

#### 4.7.1 实验目的

验证 Native、WebView、Web 三类特征各自对风险识别的贡献，证明跨端融合优于单端指纹。

#### 4.7.2 实验分组

建议至少设置以下分组：

| 分组 | 使用特征 | 目的 |
| --- | --- | --- |
| Full | Native + WebView + Web | 完整系统上限 |
| Native Only | 仅 Android 原生特征 | 验证硬件/传感器/电池贡献 |
| WebView Only | 仅宿主容器特征 | 验证 JSBridge、安装来源、WebView Provider 贡献 |
| Web Only | 仅 Web 前端特征 | 验证浏览器渲染与运行环境贡献 |
| Full - Native | 移除 Native | 观察模拟器与云真机识别能力下降 |
| Full - WebView | 移除 WebView | 观察 API 重放、容器剥离识别能力下降 |
| Full - Web | 移除 Web | 观察 Headless、WebGL、Canvas、算力异常识别能力下降 |

#### 4.7.3 预期结论

Native 特征对模拟器和云真机识别最关键；WebView 特征对 App 宿主真实性和接口重放识别最关键；Web 特征对无头浏览器、PC 环境和渲染异常识别最关键；完整特征组合应取得最低误差和最高攻击召回率。

### 4.8 App 本地评分闭环实验

#### 4.8.1 实验目的

验证轻量评分器集成到 App 后，系统能否在无后端实时参与的情况下完成采集与评分。

#### 4.8.2 实验流程

1. 启动 App，生成同一 `session_id`。
2. 完成 Native 与 WebView/Web 探针采集。
3. 端侧构造特征向量。
4. 调用 `DeviceRiskScorer.score(input)`。
5. 展示风险分数、风险等级和关键原因。

#### 4.8.3 评估内容

评估端侧推理耗时、采集完成率、异常场景容错、与 Python 侧预测结果的一致性。

### 4.9 攻击样本案例分析

#### 4.9.1 API 重放样本

分析 Native 字段缺失、JSBridge 为 false、UA 异常等特征如何导致高风险判定。

#### 4.9.2 无头 PC 浏览器样本

分析平台字段、触控点、算力耗时、WebGL Renderer 等特征的异常。

#### 4.9.3 廉价模拟器样本

分析 x86 ABI、goldfish/ranchu、传感器缺失、SwiftShader、电池温度固定等特征。

#### 4.9.4 云真机/测试机架样本

分析传感器正常但 ADB、安装来源、电量和时区等组合特征导致中风险判定的原因。

### 4.10 本章小结

总结三端采集完整性、规则知识库、三端消融实验、本地评分闭环和攻击案例分析结果，说明本文系统架构的有效性。

## 结论

### 主要工作总结

1. 设计了基于 Android Native、WebView、Web 三端融合的设备指纹采集方案。
2. 实现了 HybridGuard App、FastAPI 后端、数据合并与持久化流程。
3. 构建了包含真实样本、扩充样本和高危模拟样本的风险评分数据集。
4. 利用本地大模型分析跨层 feature 的语义对齐关系，构建规则知识库并生成风险标签。
5. 初步结果表明随机森林在当前数据集上部署路径更直接、评分误差更低，适合作为 App 本地评分组件。
6. 后续完成 App 本地评分集成与三端消融实验后，可形成完整的移动端无感风控闭环。

### 不足与展望

1. 当前规则知识库和风险标签依赖本地大模型和提示词，后续需要引入人工抽样复核和更多真实攻击样本。
2. 随机森林 Java 代码体积较大，后续可考虑模型剪枝、特征筛选或替换为更轻量的评分组件。
3. 端侧类别编码和训练侧编码需要严格同步，后续应生成统一的特征 Schema 与编码表。
4. 数据集规模仍需扩大，特别是不同品牌、不同系统版本、不同 WebView 内核和真实黑产环境样本。
5. 可进一步研究隐私保护机制，例如本地评分、脱敏上传和联邦学习式模型更新。
